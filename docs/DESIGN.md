## Problem Statement

라인교육연구소가 파트너 퍼실리테이터(라인메이트)의 프로젝트·워크숍 참여를 관리하고 월별로 인건비를 정산하는 과정이 두 개의 분리된 도구(구글 캘린더 + 월별 엑셀)에 걸쳐 있어, 매월 말 캘린더를 보며 엑셀에 옮기는 전사·합산 작업에서 시간 손실과 실수가 가장 크게 발생한다. 일정 관리 자체는 정산을 위한 데이터 입력일 뿐이므로, 진짜 해결해야 할 문제는 "예쁜 캘린더 UI"가 아니라 "엑셀 없이 끝나는 월말 정산"이다.

## What Makes This Cool

이 도구는 화려한 기능이 아니라 **두 사람의 시간을 직접 돌려준다**:
- 관리자: 매달 말 캘린더와 엑셀 사이를 오가던 수 시간이 한 번의 "월 마감" 클릭으로 줄어든다.
- 라인메이트: "내가 이번 달 몇 회 했고, 얼마 받는지" 확인이 카톡·전화 없이 바로 가능.

승인 워크플로우(라인메이트 입력 → 관리자 승인) 구조이기 때문에 "정산 시즌이 되면 엑셀에 한꺼번에 몰아넣는" 패턴 대신 **상시 데이터가 깨끗한 상태로 유지**된다는 점이 핵심.

## Constraints

- **사용자 규모:** 라인메이트 수십 명 수준(추정). 월간 이벤트 수십~수백 건 수준.
- **핵심 출력:** 기존 "(주)라인교육연구소_2026년 ○월 인건비내역_김연미.xlsx" 양식과 동일한 엑셀 파일이 한 번의 클릭으로 생성되어야 한다 — 회계·세무 흐름을 끊지 않기 위함.
- **단가 기준:** "(강사비)지급기준_라인교육연구소_2026.hwp" 문서가 진실의 원천. 시스템에 매트릭스를 미리 박지 않고, 관리자가 프로젝트별로 단가를 입력하는 방식(전제 2).
- **모바일 우선:** 라인메이트는 PC 앞에 앉아 있지 않다. 스마트폰에서 1분 안에 입력 가능해야 한다.
- **한국어 전용 UI**, 한국 시간대(KST), 원화(₩).

## Premises (모두 사용자 동의 완료)

1. 가장 큰 통증은 월말 정산이고, 일정 관리는 그 정산을 위한 데이터 입력일 뿐이다. → "예쁜 캘린더 UI"보다 "엑셀 없이 끝나는 정산 화면"이 훨씬 중요하다.
2. 라인메이트는 본인이 참여한 프로젝트만 직접 입력한다. 관리자는 (a) 입력 내용을 확인·승인하고, (b) 프로젝트별 비용을 입력하고, (c) 월별 정산을 마감한다. → 단가는 사전 매트릭스가 아니라 관리자가 프로젝트별로 그때그때 정하는 값이다.
3. 라인메이트가 입력한 참여 내역을 관리자가 승인하기 전까지는 정산에 반영되지 않는다. → 허위·중복 입력 방지.
4. "월별 인건비내역" 엑셀(이미 있는 양식)은 정산 결과의 최종 출력이다. → 시스템 안에서만 끝나는 게 아니라 기존 엑셀 양식으로 내보내기가 가능해야 한다.

## Approaches Considered

### Approach A: 구글 시트 + 폼 (최소형)
- 노력: S, 위험: Low.
- 장점: 비용 0원, 1~2일 내 시작.
- 단점: 진짜 로그인·승인 워크플로우 어려움, 데이터 늘면 깨짐.

### Approach B: 노코드 플랫폼 (Glide / Bubble 등)
- 노력: M, 위험: Med.
- 장점: 모바일·역할 분리 기성품 수준.
- 단점: 월 구독료(3~7만원), 한국어 엑셀 양식 정확 재현에 한계, 락인.

### Approach C: 라이트 커스텀 웹앱 (선택됨)
- 노력: L, 위험: Med.
- 장점: 워크플로우와 엑셀 양식에 정확히 맞춤, 운영비 1~2만원/월, 모바일 대응 자유.
- 단점: 첫 버전 시간 길고, 유지보수 책임 발생.

## Recommended Approach: C — 라이트 커스텀 웹앱

### 기술 스택 (제안)

- **프론트엔드:** Next.js (App Router) + Tailwind CSS — 모바일 반응형 기본, Vercel 무료 호스팅.
- **백엔드·DB:** Supabase (PostgreSQL + Auth + Row-Level Security) — 무료티어로 충분, 한국어 잘 처리, 인증 내장.
- **엑셀 출력:** ExcelJS — 기존 .xlsx 양식 셀 단위 재현 가능.
- **인증:** 카카오 로그인 또는 이메일 매직링크. (라인메이트가 비밀번호 잊어버리는 마찰 줄이기)

### 데이터 모델 (핵심 4개 테이블) — v2 (plan-eng-review 반영)

```
linemates      (id, name, phone, email, role_default,
                status: pending|active|inactive,          -- 자체 가입 후 관리자 승인
                created_at, approved_at, approved_by)

projects       (id, name, client, period_start, period_end,
                default_unit_price NUMERIC NOT NULL,
                notes, created_at)

participations (id, linemate_id, project_id, date, role, hours,
                unit_price NUMERIC NULL,                  -- NULL이면 project.default_unit_price 사용
                status: pending|approved|rejected,
                submitted_at, approved_at, approved_by,   -- 감사 추적
                UNIQUE(linemate_id, project_id, date))    -- 중복 제출 방지

settlements    (id, year_month CHAR(7),                  -- 예: '2026-04'
                linemate_id,
                total_amount NUMERIC,                     -- finalize 시점에만 계산·저장
                status: finalized,                        -- draft는 실시간 SUM으로 표시, 저장 안 함
                finalized_at, finalized_by, exported_at)
```

**인덱스 (마이그레이션 파일에 포함):**
```sql
CREATE INDEX idx_participations_status       ON participations(status);
CREATE INDEX idx_participations_linemate_date ON participations(linemate_id, date);
CREATE INDEX idx_participations_project_status ON participations(project_id, status);
```

**RLS 정책 (Supabase):**
```sql
-- linemates: 본인만 읽기, 쓰기는 service_role(관리자)만
-- participations: SELECT = 본인(linemate_id) 또는 관리자
--                 INSERT = 본인만 (status=pending으로 강제)
--                 UPDATE = 관리자만 (승인/거절/단가 설정)
-- settlements: SELECT = 본인 또는 관리자 / INSERT·UPDATE = 관리자만
-- projects: 모든 active 라인메이트 읽기 가능 (드롭다운용), 쓰기는 관리자만
```

**월 마감 RPC (원자성 보장):**
```sql
-- finalize_month(p_year_month TEXT) RETURNS void
-- 트랜잭션 안에서:
--   1. settlements WHERE year_month=p_year_month FOR UPDATE (잠금)
--   2. 이미 finalized면 RAISE EXCEPTION
--   3. approved participations SUM(COALESCE(unit_price, project.default_unit_price))
--   4. settlements INSERT (total_amount, status='finalized', finalized_at, finalized_by)
--   5. participations UPDATE SET locked=true WHERE year_month=p_year_month (수정 잠금)
```

핵심 룰:
- `participations.status = 'approved'` 인 것만 정산에 포함.
- 단가: `unit_price` 컬럼이 NULL이면 `projects.default_unit_price` 폴백.
- Draft 화면에서는 실시간 `SUM` 쿼리로 예상액 표시 (DB 저장 안 함).
- 월 마감은 `finalize_month` RPC로만 가능 (트랜잭션, 이중 마감 방지).
- 마감 후 해당 월 참여 내역 수정 불가 (RLS + locked 컬럼으로 이중 잠금).

### 주요 화면 (v3 — plan-design-review 반영)

**라인메이트 모드 (모바일 우선, 375~430px 기준):**

1. **로그인** → 이메일 매직링크 (또는 카카오)
2. **내 대시보드**
   - **Hero 숫자: 이번 달 예상액 ₩X** (가장 큰 글자, pending+approved 합)
   - 보조: 확정액 ₩Y (작은 글자, finalized된 직전 월)
   - 보조: 이번 달 N건 (대기 a / 승인 b / 거절 c)
   - 거절된 건 있으면 상단 알림 배지 → 탭하면 이력 화면 거절 행으로 이동
   - **빈 상태 카피:** "이번 달 입력 없음. 아래 +참여 입력으로 시작하세요."
   - **승인 대기 상태 (가입 후):** "관리자 승인 대기 중입니다 (보통 1~2일)"
3. **+참여 입력** — 폼 1개, 캘린더 UI 없음 (전제 1)
   - 필드 순서: **날짜 → 프로젝트 → 역할 → 시간**
   - 날짜: `오늘` / `어제` 칩 + 캘린더 fallback (한 번 탭으로 끝)
   - 프로젝트 드롭다운: 본인 최근 3개월 프로젝트 상단, 그 아래 전체
   - 역할: `role_default` 미리 선택, 변경 가능
   - 중복 제출 (UNIQUE 위반): "이미 입력하셨어요. 수정하시겠어요?" → 기존 행 편집 모드
   - 모든 hit target 최소 44px
4. **내 참여 이력**
   - 월별 그룹 (디폴트 이번 달), 같은 월 안에서 날짜 내림차순
   - 각 행에 상태 칩(승인=초록·✓ / 대기=노랑·⏱ / 거절=빨강·✕)
   - 거절 행 펼치면 사유 + `다시 제출` 버튼
5. **내 정산 이력** — 월별 확정 금액, 본인 분 엑셀 다운로드

**관리자 모드 (데스크톱 우선, 1280px+):**

1. **대시보드** — 2-tier 위계
   - **상단 (오늘 처리할 일):** "참여 승인 대기 N건" 큰 카드 + "신규 가입 승인 N명" 카드 (값 0이면 "처리할 일 없음 ☕")
   - **하단 (이번 달 현황):** 미정산 라인메이트 수, 월별 총 인건비 추이 (단순 막대 SVG)
2. **신규 멤버 승인** — 자체 가입자 목록 → 승인(active) / 거절(inactive)
3. **승인 큐** — 일괄 처리 우선
   - 체크박스 + `선택한 N건 승인` 버튼
   - 의심 행만 펼쳐서 단가/거절 사유 입력
   - 키보드 단축키: `j/k` 이동, `a` 승인, `r` 거절 (파워유저용)
   - 정렬: 제출일·라인메이트별·프로젝트별
   - **빈 상태:** "승인할 게 없어요 ☕"
4. **프로젝트 관리** — 프로젝트 추가·기본 단가 설정
5. **라인메이트 관리** — 목록·연락처·상태
6. **월 마감 화면** — 무게감 있는 워크플로우
   - **마감 전 체크리스트 카드:**
     - ✅ pending 0건 (남아 있으면 "N건 처리하세요" 링크)
     - ✅ 모든 라인메이트 단가 설정됨 (미설정자 클릭 가능)
   - 실시간 SUM (라인메이트별 + 총합)
   - 두 버튼: `엑셀 내보내기` (반복 가능) + `마감 확정` (1회성, 빨간 강조)
   - 마감 차단 정책: pending 0건 아니면 마감 버튼 비활성
   - **마감 후 확인 화면:** "○월 마감 완료. 총 ₩X 지급 예정. 라인메이트 N명에게 알림 발송됨." + 엑셀 다운로드 버튼
   - 되돌리기 불가 (UI에서 제공 안 함, DB 직접 수정 필요) → 마감 클릭의 무게 유지

**모바일에서 관리자 모드:** 읽기 전용 (대시보드만). 승인·마감은 PC.

### MVP 범위 (4주 안에 첫 사용 가능)

**Week 1:** 프로젝트 셋업, DB 스키마, Supabase Auth, 라인메이트 등록 화면.
**Week 2:** 라인메이트 참여 입력 화면 + 관리자 승인 큐.
**Week 3:** 월별 정산 집계 화면 + 엑셀 출력 (기존 양식 1:1 재현).
**Week 4:** 모바일 마감, 실데이터 1주일 병행 운영(엑셀과 비교 검증).

### 마이그레이션

기존 2026년 1~3월 인건비 엑셀은 시스템에 한 번에 import (관리자 모드 일회성 도구). 4월부터 신규 입력은 시스템에서.

## 알림 설계 (MVP 포함)

Supabase Database Webhooks → Edge Function 패턴:

| 이벤트 | 수신자 | 내용 |
|--------|--------|------|
| participation.status → pending | 관리자 | "○○님이 참여 내역 N건을 제출했습니다" |
| participation.status → approved | 라인메이트 | "참여 내역이 승인되었습니다 (프로젝트명, 날짜)" |
| participation.status → rejected | 라인메이트 | "참여 내역이 거절되었습니다 (사유 포함)" |
| settlements.status → finalized | 라인메이트 | "○월 정산이 확정되었습니다 (₩X원)" |
| linemates.status → active | 라인메이트 | "계정이 승인되었습니다. 지금 입력하세요" |

이메일: Supabase 내장 이메일 (무료티어 일 3건 제한 — 초과 시 Resend.com 연동, 월 $0~$20).

## Design System (v3 추가)

신규 프로젝트 — 처음부터 시작. 최소 토큰만 정의:

| 토큰 | 값 |
|------|------|
| 폰트 | Pretendard (한국어 가독성·무료·웹폰트) |
| 본문 크기 | 16px / 줄간격 1.6 |
| 숫자 강조 | tabular-nums + ₩ 기호 (₩1,234,567) |
| 색상 | 파란색 1색 (primary) + 그레이 6단계 |
| 상태 색 | 승인=초록·✓ / 대기=노랑·⏱ / 거절=빨강·✕ (색+아이콘+텍스트 3중) |
| 모바일 hit target | 최소 44px |
| 차트 | 단순 SVG 막대 (라이브러리 풀 도입 X) |

**금지:**
- 캘린더 UI를 "예쁘게" 만들려는 시도 (전제 1에서 거부)
- Recharts 등 차트 라이브러리 풀 도입 (1개 차트에 과투자)
- 영어 직역 빈 상태 카피 ("Looks like..." 패턴 금지)

## 보안 & 컴플라이언스 (v4 — /cso 반영)

이 시스템은 **개인정보(이름·전화·이메일)** + **소득정보(정산액·단가)** 를 다룹니다. 한국 PIPA(개인정보보호법) 적용 대상 — 위반 시 매출 3% 과징금. 아래 8개 HIGH 결정 사항은 코드 작성 전 확정.

### 1. RLS 정책 SQL 풀 텍스트 (자연어 → 실제 SQL)

```sql
-- linemates ─────────────────────────────────────────────────────
ALTER TABLE linemates ENABLE ROW LEVEL SECURITY;

-- SELECT: 본인 행 OR 관리자
CREATE POLICY linemates_select ON linemates FOR SELECT
  USING (
    id = auth.uid()
    OR (auth.jwt() -> 'app_metadata' ->> 'role') = 'admin'
  );

-- INSERT: anon이 자체 가입 (status=pending 강제)
CREATE POLICY linemates_insert_self ON linemates FOR INSERT
  WITH CHECK (
    id = auth.uid()
    AND status = 'pending'
  );

-- UPDATE: 관리자만 (status·승인 정보 변경)
CREATE POLICY linemates_update_admin ON linemates FOR UPDATE
  USING ((auth.jwt() -> 'app_metadata' ->> 'role') = 'admin');

-- participations ─────────────────────────────────────────────────
ALTER TABLE participations ENABLE ROW LEVEL SECURITY;

-- SELECT: 본인 OR 관리자
CREATE POLICY participations_select ON participations FOR SELECT
  USING (
    linemate_id = auth.uid()
    OR (auth.jwt() -> 'app_metadata' ->> 'role') = 'admin'
  );

-- INSERT: 본인만, status=pending 강제, active 라인메이트만
CREATE POLICY participations_insert_self ON participations FOR INSERT
  WITH CHECK (
    linemate_id = auth.uid()
    AND status = 'pending'
    AND EXISTS (
      SELECT 1 FROM linemates
      WHERE id = auth.uid() AND status = 'active'
    )
  );

-- UPDATE: 관리자만 (승인/거절/단가 보정), 마감된 건은 차단
CREATE POLICY participations_update_admin ON participations FOR UPDATE
  USING (
    (auth.jwt() -> 'app_metadata' ->> 'role') = 'admin'
    AND locked = false
  );

-- settlements ────────────────────────────────────────────────────
ALTER TABLE settlements ENABLE ROW LEVEL SECURITY;

-- SELECT: 본인 OR 관리자
CREATE POLICY settlements_select ON settlements FOR SELECT
  USING (
    linemate_id = auth.uid()
    OR (auth.jwt() -> 'app_metadata' ->> 'role') = 'admin'
  );

-- INSERT/UPDATE: 관리자만 (실제로는 RPC finalize_month 통해서만)
CREATE POLICY settlements_write_admin ON settlements FOR ALL
  USING ((auth.jwt() -> 'app_metadata' ->> 'role') = 'admin');

-- projects ───────────────────────────────────────────────────────
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;

-- SELECT: 모든 active 라인메이트 (드롭다운 필요)
CREATE POLICY projects_select ON projects FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM linemates
      WHERE id = auth.uid() AND status = 'active'
    )
    OR (auth.jwt() -> 'app_metadata' ->> 'role') = 'admin'
  );

-- INSERT/UPDATE: 관리자만
CREATE POLICY projects_write_admin ON projects FOR ALL
  USING ((auth.jwt() -> 'app_metadata' ->> 'role') = 'admin');
```

### 2. role 컬럼 위치 — `auth.users.app_metadata` 사용 (사용자 변경 불가)

- ❌ **하지 않을 것:** `linemates.role` 컬럼에 'admin'/'linemate' 저장
- ✅ **할 것:** Supabase `auth.users.app_metadata.role` 에 저장 (raw JWT 클레임 — 사용자가 절대 수정 불가)
- 관리자 지정 시 SQL: `UPDATE auth.users SET raw_app_meta_data = raw_app_meta_data || '{"role":"admin"}' WHERE id=...;` (Supabase Studio에서 수동 또는 CLI)
- RLS 정책에서 `auth.jwt() -> 'app_metadata' ->> 'role'` 로 읽음

### 3. pending 라인메이트의 데이터 접근 차단

- 가입 직후 status=pending — 관리자 승인 전까지는:
  - 본인 `linemates` 행만 SELECT (다른 라인메이트 명단 안 보임)
  - `participations`, `projects`, `settlements` 모두 SELECT 0건
- RLS 정책에 `EXISTS (SELECT 1 FROM linemates WHERE id=auth.uid() AND status='active')` 가드 (위 SQL에 이미 포함)
- 화면: pending 사용자는 "관리자 승인 대기 중" 페이지로 라우팅, 다른 화면 접근 차단

### 4. 가입 게이트 — 이메일 도메인 화이트리스트 + 관리자 승인 (이중 잠금)

- Supabase Auth 가입 시 이메일 도메인 검증 (Edge Function pre-signup hook 또는 trigger)
- 허용 도메인: `@lineedu.kr` + 관리자가 사전 등록한 외부 도메인 (`allowed_signup_domains` 테이블)
- 도메인 통과해도 status=pending → 관리자 승인 필요
- IP 단위 rate limit: 같은 IP에서 시간당 가입 시도 5회 초과 차단 (Supabase 또는 Vercel 미들웨어)

### 5. service_role 키 격리 정책

- ❌ **절대 하지 않을 것:**
  - Vercel 환경변수에 `SUPABASE_SERVICE_ROLE_KEY` 저장
  - Next.js Server Action·Server Component에서 service_role 사용
  - 클라이언트 번들에 노출 (`NEXT_PUBLIC_*` 접두사 절대 X)
- ✅ **할 것:**
  - service_role 키는 Supabase Edge Function 환경변수에만
  - 클라이언트(브라우저)는 anon key + 사용자 JWT만 사용 → RLS가 검증
  - 서버에서 RLS 우회 작업 필요 시 Edge Function 호출 (예: finalize_month RPC)
- Vercel env 검사: 빌드 시 `SUPABASE_SERVICE_ROLE_KEY` 변수 존재하면 빌드 실패시키는 가드 추가

### 7. Edge Function webhook secret 검증

Database Webhook → Edge Function 호출 시 위조 방지:

```typescript
// supabase/functions/notify/index.ts (예시)
import { serve } from "https://deno.land/std/http/server.ts"

serve(async (req) => {
  const secret = req.headers.get('x-webhook-secret')
  if (secret !== Deno.env.get('WEBHOOK_SECRET')) {
    return new Response('Unauthorized', { status: 401 })
  }
  // ... 알림 로직
})
```

- `WEBHOOK_SECRET` 은 Supabase Dashboard에서 webhook 설정 시 헤더로 추가 (서명 검증용)
- Edge Function 환경변수에도 동일 값 저장
- 외부에서 Function URL을 알아도 secret 없으면 401

### 8. 마감 잠금 3중 방어

원자성 보장 + 사고 방지:

1. **RPC `finalize_month` 트랜잭션** — 이미 정의됨 (도큐먼트 기존 섹션)
2. **`participations.locked = true` 컬럼** — finalize 시점에 일괄 UPDATE
3. **Postgres trigger** (신규):
   ```sql
   CREATE OR REPLACE FUNCTION prevent_locked_update() RETURNS trigger AS $$
   BEGIN
     IF OLD.locked = true THEN
       RAISE EXCEPTION '마감된 월의 참여 내역은 수정 불가 (id=%, 월=%)',
         OLD.id, to_char(OLD.date, 'YYYY-MM');
     END IF;
     RETURN NEW;
   END;
   $$ LANGUAGE plpgsql;

   CREATE TRIGGER prevent_locked_participations
     BEFORE UPDATE OR DELETE ON participations
     FOR EACH ROW EXECUTE FUNCTION prevent_locked_update();
   ```
4. RLS UPDATE 정책에도 `locked = false` 조건 (위 #1 SQL에 이미 포함)

## What I noticed about how you think

- 통증을 정확히 짚으셨다. "월말 정산 계산"이라고 답하신 순간, 도구 전체의 우선순위가 명확해졌다. "캘린더 UI를 예쁘게"라고 답하는 사람과 다른 길이다.
- 권한 설계 질문에 "기타"로 직접 풀어 쓰신 답("라인메이트는 입력만, 관리자는 확인·단가입력·정산")이 시스템 핵심 흐름 그 자체였다. 이미 운영을 머릿속에서 충분히 시뮬레이션해 본 사람의 답이다.
- 단가 규칙에서 "2번과 3번 둘 다"라고 답하신 부분은 일반적이지 않다. 보통은 단순화하려고 한쪽을 고른다. 현실의 복잡성을 그대로 인정하는 태도가 시스템을 깨지지 않게 만든다.
